Labrador Governance

Goldador is the shared source of truth for Labrador membership, team access, and leadership permissions. Use this page to understand what access each role receives before you register or update your record.

Connected resources

GitHub
Code repositories, project teams, reviews, and issue tracking.
Google Drive

OpenBao

Role Permissions

Roles build on each other. Team and leadership roles add project or committee access on top of basic Labrador membership.

Base access

Member

For anyone officially registered as part of Labrador.

  • GitHub organization member access.
  • ScottyLabs Google Group member access.
  • ScottyLabs Google Drive contributor access.
Project access

Team Member

For contributors listed on a Labrador project team.

  • Membership in the team’s GitHub team.
  • Write access to the team’s GitHub repositories.
  • Membership in the team’s Slack channel.
  • Team app login through Keycloak.
  • Read/list access to local project secrets in OpenBao.
Project admin

Team Lead

For team leads responsible for project access and ownership.

  • Admin access to the team’s GitHub repositories.
  • CODEOWNERS review ownership for the team’s Goldador team file, meaning team leads review membership and permission changes before they are applied.
  • Create, read, update, delete, list, and sudo access to the team’s project secrets in OpenBao.
Committee access

Leadership Member

For Labrador leadership members who need committee-wide visibility.

  • Keycloak read-only access.
  • Committee-wide create, read, update, delete, list, and sudo access to Labrador secrets in OpenBao.

Leadership membership is separate from project team leadership.

Committee admin

Leadership Lead/Admin

For leadership leads who administer Labrador-wide access.

  • GitHub organization admin access.
  • ScottyLabs Google Group manager and member access.
  • ScottyLabs Google Drive content manager access.
  • Keycloak admin access.
  • Admin-level secrets access in OpenBao.
  • Default CODEOWNERS review ownership for Goldador and the teams/ directory.

Register or Update Access

Goldador changes happen through pull requests. Pick the smallest change that represents the access you need, then let validation and review confirm it.

Prerequisites

Link Your Accounts

Goldador validates identity links before it grants Labrador access.

  • Create a Keycloak account .
  • Link Keycloak to your GitHub account.
  • Link Keycloak to your Slack account.
Setup

Prepare a Pull Request

Work from your fork, then submit the change back to Goldador.

  • Fork the Goldador repository .
  • Clone your fork locally, following the GitHub cloning guide if needed.
  • Install the recommended editor extensions so TOML files auto-format.
Members

Join Labrador

Add yourself as a member before you are listed on a project team.

  • Create members/<github-username>.toml.
  • Add your full name and Andrew ID.
  • Submit the pull request from your own GitHub account.
  • Follow the member schema documentation for field details.
Teams

Join or Register a Team

Team leads maintain the project record that drives repository, Slack, identity, and secrets access. Members can join one team per pull request.

  • Add yourself to the team you want to join in teams/.
  • Team leads can create or update teams/<team-slug>.toml.
  • Use a short, lowercase, hyphen-separated slug for new teams.
  • List leads, members, repositories, and app URLs when they are ready.
  • Follow the team schema documentation for field details.
Validation

Pass the Checks

CI validates formatting, schemas, and Goldador's access rules before review.

  • Member and team TOML must match the published schema.
  • New team members must already exist in members/.
  • Keep each pull request focused on one member or one team change.
  • A reviewer approves CI after checking that the pull request contains no malicious code.
  • See the validator documentation for local validation details.
Review

Review and Merge

Maintainers and CODEOWNERS review access changes before they become active.

  • Reviewers are requested automatically from the CODEOWNERS file .
  • A reviewer may convert the pull request to draft while changes are needed.
  • Address reviewer comments, keep the branch up to date, and mark it ready again.
  • Reviewers squash-merge approved pull requests after all checks pass.
  • Typical squash titles look like feat(member): add Yuxiang to leadership team or feat(team): update description of leadership team.
  • After merge, synchronizers apply the approved access model.

Ready to make a change?

Open the Goldador repository, edit the relevant TOML file, and submit a pull request for review.
